## Goal
Fix the "Puzzle Tower starts in Game Over" bug and extend DreamPlay Park with three new canvas-based attractions — **Flappy Wings** (Flappy Bird style), **Tank Battlegrounds** (Tank War style), and **Sky Squadron** (Shooting Plane style) — wired through the existing registry, park map, and persistence layers.

## Context

- **Stack**: React 18 + TypeScript (strict) + Vite 5, Tailwind 3, React Router 6, Zustand (persist), Framer Motion, Canvas + `requestAnimationFrame`. No backend, no new dependencies are needed.
- **Game contract** (from `src/games/registry.ts`): every game exports a default React component matching `GameComponentProps` and reports state through `onScore` and `onGameOver` callbacks. The shell (`GameShell`) reads `restartKey`, `isPaused`, persists high scores via the store, and mounts the canvas inside a `requestAnimationFrame` loop driven by the shared `useGameLoop` and `useKeyboard` hooks.
- **Park map layout** (`src/pages/ParkPage/attractionLayout.ts`) hard-codes a 3-column × 2-row grid with the bottom-right cell spanning both rows. Adding 3 more games means the grid must be reshaped to fit 8 cards cleanly on desktop while keeping the mobile single-column stack intact.
- **Bug source** (`src/games/tetris/useTetris.ts`): the initial `createInitialState` sets `status: 'ready'`. The first call to `step()` invokes `tryMove(s, 0, 1)`, but `tryMove` is gated on `state.status !== 'running'` and returns the *same* state object unchanged. The equality check `moved === s` therefore succeeds and `lockAndAdvance(s, s.active)` is invoked, which:
  1. Stamps the spawn piece's visible cells (those with absolute `y >= 0`) into row 0 of the board (e.g. a T piece at `SPAWN_Y = -1` writes three cells at columns 3, 4, 5).
  2. Calls `popNext` to create the next piece at `(SPAWN_X, -1)`.
  3. Runs the collision check. The new piece's cells at absolute `y = 0` overlap the just-locked cells, so `collides` returns `true`, and the function returns `{ ...withNext, status: 'gameover' }`.
  4. The `useEffect` watcher then fires `onGameOver(0)` and the Game Over overlay appears on the very first gravity tick — matching the reported "goes to Game Over directly" symptom.
- **Achievements**: the existing `park-explorer` achievement still tracks only the original 5 games; it will stay unchanged. New per-game score achievements can be added later in a follow-up plan.

## Implementation Notes

### Bug fix — Puzzle Tower (`src/games/tetris/`)
1. **Root cause** is in `createInitialState` (`useTetris.ts:118-131`): it returns `status: 'ready'`. The `step()` callback (`useTetris.ts:201-212`) calls `tryMove`, which gates on `state.status !== 'running'`, returns the *same* state object, and the subsequent `moved === s` check triggers `lockAndAdvance`. Because the spawn piece is at `SPAWN_Y = -1`, `lockAndAdvance` stamps its visible cells (absolute y ≥ 0) into row 0, then `popNext` collides with them and flips status to `gameover`.
2. **Fix** (smallest correct change): change the initial status from `'ready'` to `'running'` in `createInitialState`. The `'ready'` status is not used by any UI gate, the canvas already draws the active piece, and `useGameLoop` already pauses on `isPaused || state.status === 'gameover'`. Removing the dead state also lets us drop the `state.status === 'ready'` reset branch in the game-over effect.
3. **Optional cleanup**: if `'ready'` is removed from `TetrisStatus`, the `useEffect` reset branch can be removed too (the `restartKey` re-mount zeroes everything).
4. **Test by hand**: open `/play/tetris`, verify the piece falls on the first gravity tick (no game over overlay) and the scoring / line clearing still work.

### New game folder pattern (template)
For each of `flappy-bird/`, `tank-war/`, `shooting-plane/`:
```
src/games/<gameId>/
  <Name>Game.tsx        # default export, matches GameComponentProps
  use<Name>.ts          # pure logic hook (state, step, controls)
  types.ts              # local state types
  constants.ts          # canvas size, gravity, colors, scores
  index.ts              # barrel: re-exports the component + hook
```
The component must:
- Accept `gameId`, `isPaused`, `restartKey`, `onScore`, `onGameOver`, `onRestart` from `GameComponentProps`.
- Call `useGameLoop` with `isPaused || state.status === 'gameover'` as the pause flag.
- Use `useKeyboard` (or direct `keydown` listeners for held-key loops, mirroring `TetrisGame.tsx`) for controls.
- Redraw the canvas in a `useEffect` on every state change.
- Show a Game Over overlay that calls `onRestart` when the user clicks "Play Again".
- Call `useGameStore.getState().markGamePlayed(gameId)` on the first game-over of a run.

- Collision: bullet ↔ enemy, enemy ↔ player.
- Score: +10 per enemy destroyed.
- Controls: arrow keys (or WASD) for movement, Space to fire.

### Risks & mitigations
- **Layout regression on small viewports**: keep the `md:hidden` mobile stack; the 4×2 grid only applies at the `md` breakpoint and above. Verify with a 1024×768 desktop window and a 360-wide mobile emulator.
- **Bundle bloat**: each new game adds ~5–10 kB minified; total project should still be well under the 500 kB gzipped budget noted in the README. `npm run build` prints the final size; if any new game pushes it close, lazy-import the game module through a `React.lazy` wrapper in `GamePage` (out of scope for this plan unless observed).
- **Pause / restart regressions**: every new game must honor `isPaused` (skip ticks) and `restartKey` (full re-mount via `key={restartKey}` in the shell — already handled).
- **No new top-level dependencies**: if a game really needs something, do not add it; the existing `utils/sound` WebAudio helpers are sufficient.
- **Color contrast**: pull pipe / enemy / bullet fills from the existing palette documented in `tailwind.config.ts` and use a dark stroke to keep contrast on the `night-deep` background.
